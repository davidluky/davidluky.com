# Retrospective — 2026-04-30

## Context

Full Mode audit and hardening session for `davidluky.com`, focused on making the site more secure, current, documented, maintainable, and deployment-ready.

## What Went Well

- Parallel read-only audit lanes found the real risk areas quickly: hardcoded eBay token, blind eBay POST acknowledgement, stale project data, weak CI gates, and old docs.
- Root-cause fixes landed instead of one-off patches:
  - eBay secrets moved to Worker bindings.
  - eBay POSTs now require signed notification verification.
  - project data now has typed status/visibility/live URL fields.
  - CI now runs secret scanning, build checks, site validation, and audit gates before deploy.
- The custom site validator immediately proved useful by catching its own mojibake-pattern mistake, then became a lasting guard for links, JSON-LD, headers, mojibake, and secret regressions.
- The second review pass caught smaller but important polish issues: disclosure semantics, live-only JSON-LD, and a more realistic gitleaks regex.
- Verification ended clean:
  - `npm run verify`
  - `npm audit --audit-level=moderate`
  - `npx wrangler deploy --dry-run --outdir .wrangler-dry-run`
  - local HTTP smoke tests

## What Was Risky

- The untracked `src/worker.ts` already contained production-shaped secret material. Even though it was not committed, it was close enough to deploy shape that it needed immediate structural correction.
- eBay signature verification could be implemented only to the contract level locally. True end-to-end verification still depends on real eBay credentials and a sandbox/production signed notification.
- The project refresh touched many docs and content files, increasing the chance of stale documentation or accidental scope drift.
- Replacing docs wholesale made the current contract clearer, but it also means old narrative detail is now mostly preserved only in historical audit/session files.

## Lessons

- Any new Worker/API route should start with an explicit env binding interface and a no-secrets-in-source guard.
- A single `link` field is too weak for portfolio data. Public URLs, repo URLs, internal dashboards, and private tools need different fields.
- CI needs project-specific guards in addition to generic build/type checks. The things that actually regress here are content shape, links, language copy, CSP, and secrets.
- Accessibility fixes should prefer native semantics first. `<details>/<summary>` was a better fit than focusable fake-button stat tiles.
- Dependency overrides should be checked against tool compatibility. The `yaml` override fixed the audit finding; the Vite override fixed Astro's runtime warning.

- Full Mode worked well here because the task genuinely benefited from parallel audit lanes and a second review pass.
- The strongest pattern was: audit first, fix root cause, add guard, update docs, then verify.
