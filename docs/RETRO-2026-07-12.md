# Retrospective — Matheus photobook and password gate (2026-07-12)

## Context

This project added a server-side password gate to `matheus.davidluky.com`, built a new eight-chapter
photography-book edition, preserved the generated manual and magazine editions, and deployed the
result through the existing Cloudflare Worker pipeline.

The final information architecture is additive:

- `/` — authenticated selector with three editions.
- `/livro/` — original generated book/manual.
- `/revista/` — generated editorial magazine.
- `/fotolivro` — new hand-authored photography book.

The repository is public. Real passwords, session secrets, cookies, and login submissions remained
outside tracked files and commit history.

## Delivered outcome and evidence

- The gate is enforced by the Worker before Static Assets for every Matheus-host request.
- Missing gate secrets fail closed with `503`; content is never served without the gate.
- Signed sessions use a hardened host-only cookie with `HttpOnly`, `Secure`, and `SameSite=Lax`.
- The main-host `/matheus/*` bypass redirects to the gated HTTPS subdomain.
- Unknown and future asset paths are gated before the nearest Matheus `404` can render.
- The selector exposes exactly the manual, magazine, and photography-book editions.
- Moving the photobook from `/` to `/fotolivro` preserved its prior file bytes.
- `public/matheus/livro/` and `public/matheus/revista/` retained their starting Git tree identities.
- The final suite contained 3 test files and 37 passing tests.
- `npm run verify`, the Wrangler dry run, local authenticated smoke checks, GitHub quality checks,
  Cloudflare deployment, and production guest/authenticated smoke checks passed.

## What went well

### The plan made the important invariants explicit

The spec and ten-task plan established the public-repository secret boundary, required local and
production verification, preserved the two generated editions, and separated pure session primitives
from Worker routing. Small commits made the implementation understandable and recoverable.

### Security was verified as behavior, not inferred from code

Tests covered password comparison, session signing and expiry, cookie parsing, safe return paths,
missing-secret failure, protected assets, unknown paths, nested `404` behavior, main-host redirects,
and unrelated eBay/main-site regressions. Production checks then proved the same boundary on the real
Cloudflare route.

### The original editions had an objective preservation check

Capturing the starting Git tree IDs for `livro/` and `revista/` turned “do not change these” into a
machine-checkable invariant. That was much stronger than relying on a visual diff or file counts.

### The post-launch correction stayed small

The selector was restored without rewriting the photobook's hundreds of relative image, font,
`srcset`, fragment, and edition links. Keeping the photobook as the flat file
`public/matheus/fotolivro.html` allowed the clean `/fotolivro` route while preserving its bytes and
asset base.

### “Live” meant production truth

The work was not called live after `git push`. The exact GitHub Actions run was monitored until both
quality and deploy completed, then guest and authenticated requests were checked against production.

## What was difficult or risky

### 1. The landing-page choice was technically specified but not truly settled

The original design decision replaced the familiar selector with the photobook at `/`. That matched
the written plan, but David's immediate post-launch preference was to keep the selector and make the
new edition a third option.

The mistake was not failure to follow the spec; it was treating a prominent, user-facing navigation
decision as implementation detail. The better default for a new edition was additive: preserve the
existing choice, add the photobook, and only remove the selector after explicit visual approval.

### 2. Static Assets routing was initially an incomplete security boundary

A selective `run_worker_first` path list looked sufficient but could not cover unknown or newly added
paths. Cloudflare Static Assets could handle those requests before the hostname-aware Worker gate.
The final fix was `run_worker_first = true`, backed by a test that reads the Wrangler configuration.

Lesson: when access control depends on hostname while content shares one asset namespace, the Worker
must receive the entire request plane. A list of known protected paths is not a safe perimeter.

### 3. Platform-aware `404` behavior mattered

The first Worker fallback fetched `/404.html` after Assets returned a `404`. With
`not_found_handling = "404-page"`, Assets had already chosen the nearest nested page. Fetching again
was redundant and could select the wrong fallback. Preserving the first response—and testing that only
one asset fetch occurs—was the correct model.

### 4. Secret setup needed a stricter parser

An early setup attempt used a permissive match against a Markdown instruction file and could not prove
that it had selected the intended session-secret representation. No literal entered Git or captured
output, but ambiguity is enough to treat secret provenance as unsafe. The session secret was rotated,
the ignored local state and Worker binding were synchronized, and authentication was retested.

Next time, secret automation must accept exactly one structured assignment, reject ambiguous matches,
and report only success/failure and binding names. If provenance is uncertain, rotate immediately.

### 5. Local host-aware testing had a false start

Sending a `Host` header to a Worker listening on `127.0.0.1` still exercised the main-site branch in
the current Wrangler setup. The reliable command used `--local-upstream matheus.localhost`, which made
the Worker's request URL hostname match the gated host.

Lesson: a `200` alone is not evidence that the intended hostname branch ran. Assert the redirect,
response content, and internal asset path.

### 6. Visual acceptance was weaker than the HTTP acceptance

The available browser tool could not complete the intended interactive visual pass. HTTP, content,
route, and structure checks were strong, but they could not answer whether the landing experience was
the preferred one. The selector correction exposed that gap.

For future high-salience UI changes, a real desktop/mobile preview and explicit acceptance checkpoint
must occur before production, even when all automated checks are green.

### 7. The plan was not maintained as an execution record

The implementation commits and tests prove the delivered work, but the source plan's checkboxes were
not updated as tasks landed. That leaves later readers unable to distinguish completed, changed, and
deferred steps from the plan alone.

Future plan execution should maintain checkbox state or add a short deviation log at the end of the
plan. In this run, the selector decision changed and optional Task 10 remained deferred; those facts
should be obvious without reconstructing Git history.

## Technical learnings to retain

- `run_worker_first = true` is a security control, not an optimization toggle.
- Gate unknown paths as well as known pages and assets; otherwise future content can become a bypass.
- Preserve the nested Assets `404` response instead of manually fetching a second fallback.
- A flat `/fotolivro.html` file behind a Worker mapping to `/fotolivro` keeps the existing relative
  URLs resolving from the site root. A trailing-slash route would change the base and break them.
- Rotating the password does not invalidate existing signed sessions. Rotate
  `MATHEUS_SESSION_SECRET` when all current sessions must be revoked.
- Image regeneration is not self-cleaning and uses first-source-wins for duplicate basenames. Reconcile
  input basenames, output counts, collisions, and stale derivatives before committing a refresh.
- `run_worker_first = true` increases Worker blast radius, so eBay behavior and normal main-site asset
  pass-through remain mandatory regression tests.
- Tests should lock provider configuration as well as application functions when security depends on
  both.

## What I learned about collaborating with David

- “Available” means visible as a clear choice, not merely reachable from a colophon link.
- A familiar navigation surface should be preserved unless its removal is explicitly accepted.
- “Push live” means monitor the deployment and verify production, not just push a branch.
- Secret safety includes logs, commands, screenshots, commit messages, and provenance—not only source
  files.
- Exact preservation requests benefit from hashes and tree IDs, not assurances.

## Operating changes for future work

### Before implementation

1. Identify user-visible decisions that materially change navigation or the first screen.
2. Confirm those decisions explicitly or choose the additive/reversible option.
3. Record immutable paths and their starting Git tree IDs.
4. Map provider routing order, fallback behavior, and hostname handling before writing the gate.
5. Define production evidence and rollback criteria before editing.

### During implementation

1. Keep secrets in ignored structured storage and in-memory operations only.
2. Add tests for configuration assumptions, unknown paths, and unrelated Worker branches.
3. Update operational docs when the implementation deviates from the plan.
4. Use focused commits and run the full verification immediately before each commit.
5. Pause for desktop/mobile visual acceptance before replacing an established entry point.

### Before and after deployment

1. Run the complete verification, Worker dry run, immutable-tree comparison, and local guest/auth smoke.
2. Reconfirm that staged files and commit messages contain no secret material.
3. Obtain explicit production approval for the exact final commit set.
4. Monitor the exact pushed SHA through quality and deploy.
5. Test guest and authenticated behavior on production before declaring the change live.
6. Record deviations, deferred work, the deployment run, and any new provider gotchas.

## Deferred or deliberate non-changes

- The optional cross-site Task 10 work was not folded into this security/content deployment.
- The portfolio screenshot may not yet show the final three-card selector and can be refreshed as a
  separate visual-only task.
- The generated `livro/` and `revista/` editions were intentionally not reformatted or cleaned up.
- The accepted lightweight threat model remains: one shared password, no per-user accounts or
  revocation, no application-level rate limiter, and no logout route. Global session revocation uses
  the session-secret rotation procedure.
- No secret literal belongs in this retrospective or any future tracked documentation.
