## 1. XSS via innerHTML in applyI18n()

**Severity**: Critical (security)

**What was the issue**: The `applyI18n()` function in `src/i18n/shared.ts` used `el.innerHTML = val` for every translation string. This means any translation value — including ones that don't contain HTML — was injected as raw HTML. If a translation string were ever populated from user input or a compromised data source, it would execute arbitrary scripts.

**Why it was an issue**: innerHTML parses its input as HTML, which means `<script>` tags, event handlers (`onerror`, `onclick`), and other injection vectors are all live. Even though the current translations are hardcoded, the pattern itself is dangerous: any future change that introduces dynamic content into translations would create an XSS vulnerability with no warning.

**How it was solved**: Changed `applyI18n()` to use `textContent` by default (which treats everything as plain text, no HTML parsing). For the ~7 elements that genuinely need HTML (links, `<em>` tags, `&larr;` entities), added a `data-i18n-html` attribute as an explicit opt-in. The function now checks: `if (el.hasAttribute('data-i18n-html')) el.innerHTML = val; else el.textContent = val;`

**How it is now**: Safe by default. Only elements explicitly marked with `data-i18n-html` use innerHTML. All 5 pages updated with the attribute where needed.

**Files changed**: `src/i18n/shared.ts`, `src/pages/index.astro`, `src/pages/gaming.astro`, `src/pages/about.astro`, `src/pages/404.astro`

## 2. CSP Missing base-uri and form-action

**Severity**: Critical (security)

**What was the issue**: The Content-Security-Policy header in `public/_headers` didn't include `base-uri` or `form-action` directives.

**Why it was an issue**: Without `base-uri`, an attacker who can inject a `<base href="https://evil.com">` tag can redirect all relative URLs (links, form actions, script sources) to their server. Without `form-action`, any form on the page can submit data to an arbitrary URL. Both are standard CSP hardening directives recommended by OWASP.

**How it was solved**: Added `base-uri 'self'; form-action 'self'` to the existing CSP header.

**How it is now**: CSP covers all standard directives. Forms can only submit to the same origin. Base tags can only point to the same origin.

**Files changed**: `public/_headers`

## 3. npm Vulnerabilities (Astro + PostCSS CVEs)

**Severity**: Critical (security)

**What was the issue**: `npm audit` reported 7 vulnerabilities — including CVEs in Astro itself and in PostCSS (XSS vector).

**Why it was an issue**: The Astro CVE could potentially affect build-time processing, and PostCSS XSS could affect CSS processing. Even though this is a static site (no runtime server), build-time vulnerabilities can affect the generated output.

**How it was solved**: `npm audit fix` resolved the Astro and PostCSS vulnerabilities by updating to patched versions. The remaining 5 vulnerabilities are all in the `yaml` package deep inside `@astrojs/check` (a dev-only TypeScript checker). Fixing those requires a breaking major version bump of `@astrojs/check`.

**How it is now**: Astro and PostCSS are patched. 5 remaining vulnerabilities are dev-only (never reach production output), accepted as known risk until `@astrojs/check` releases a compatible update.

**Files changed**: `package-lock.json`

## 4. (window as any) Type Casts

**Severity**: High (type safety)

**What was the issue**: Three pages used `(window as any).__stats`, `(window as any).__gaming`, and `(window as any).__about` to access server data passed via Astro's `define:vars` bridge.

**Why it was an issue**: `as any` completely disables TypeScript checking. If a property was renamed in the `define:vars` block but not in the consumer, or if the shape changed, TypeScript wouldn't catch it. It also makes refactoring risky — you can't "Find All References" on an `any`-typed property.

**How it was solved**: Created `src/types/global.d.ts` with a proper Window interface augmentation declaring all three `__stats`, `__gaming`, and `__about` properties with their exact shapes. Removed all `(window as any)` casts from the three pages.

**How it is now**: All window bridge properties are fully typed. TypeScript will catch shape mismatches between `define:vars` and consumer scripts at check time.

**Files changed**: `src/types/global.d.ts` (new), `src/pages/index.astro`, `src/pages/gaming.astro`, `src/pages/about.astro`

## 5. stats.ts Mixed Types (strings vs numbers)

**Severity**: High (data integrity)

**What was the issue**: `stats.ts` had some values as strings (e.g., `xboxGamerscore: "75,268"`) and others as numbers (e.g., `theRoomGames: 14`). Consumers couldn't predict whether they'd get a string or number.

**Why it was an issue**: Mixed types mean every consumer has to guess: do I format this, or is it pre-formatted? If a number-consumer gets a string, `toLocaleString()` fails silently. If a string-consumer gets a number, it displays without formatting. This is a class of bug that repeats every time a new stat is added.

**How it was solved**: Converted all values to numbers. Formatting (commas, locale) now happens at the template layer using `toLocaleString('en-US')` / `toLocaleString('pt-BR')`.

**How it is now**: `stats.ts` exports only numbers. Each page's template handles its own locale-aware formatting. Adding a new stat is unambiguous: always a number.

**Files changed**: `src/data/stats.ts`, `src/pages/gaming.astro`

## 7. DB Query Results Typed as `unknown`

**Severity**: High (type safety)

**What was the issue**: After adding `@types/better-sqlite3`, the `.get()` and `.all()` methods on prepared statements correctly returned `unknown` instead of the previous implicit `any`. This exposed 10 TypeScript errors where query results were accessed without type assertions (e.g., `.get().c`, `.all().map(r => r.platform)`).

**Why it was an issue**: Without type assertions, TypeScript can't verify that the accessed properties actually exist on the query result. This is exactly the kind of thing TypeScript is supposed to catch — a query could return different columns than expected, and the error would only show up at runtime.

**How it was solved**: Added inline type assertions to all 6 query sites: `.get() as { c: number }`, `.all() as { platform: string; t: number }[]`, etc. Each assertion matches the exact columns selected in the SQL query. Removed the `(r: any)` parameter annotations from `.map()` callbacks since the array type now flows through.

**How it is now**: All DB queries have explicit type contracts. If the SQL changes but the type assertion doesn't, TypeScript still checks the consumer code against the declared shape. Not perfect (the assertion itself isn't validated against SQL), but much better than `any`.

**Files changed**: `src/data/gaming.ts`

## 8. better-sqlite3 in Wrong Dependency Group

**Severity**: High (build correctness)

**What was the issue**: `better-sqlite3` was listed under `dependencies` in `package.json`, and `@types/better-sqlite3` wasn't installed at all.

**Why it was an issue**: `better-sqlite3` is a native C++ addon used only at build time (Astro's static generation reads the Game Library DB during build, not at runtime). Listing it in `dependencies` means production installs download and compile a native module that's never used. Missing `@types/better-sqlite3` meant `.get()` and `.all()` returned `any`, hiding type errors.

**How it was solved**: Moved `better-sqlite3` to `devDependencies`. Installed `@types/better-sqlite3` as a dev dependency.

**How it is now**: Only dev/build environments install the native module. Types are properly available for all DB operations.

**Files changed**: `package.json`

## 10. Silent Error Swallowing in Steam API Fallback

**Severity**: High (observability)

**What was the issue**: The `fromSteamAPI()` catch block had an empty error handler — `catch { return null; }`. If the Steam API call failed, there was no indication of what went wrong.

**Why it was an issue**: When debugging "why is the gaming page showing fallback data?", there's no log to tell you whether the DB failed, the API failed, or both. Silent failures make debugging exponentially harder.

**How it is now**: Both fallback paths log their errors during build, making it clear which data source failed and why.

## 11. Heading Hierarchy (h1 → h3, skipping h2)

**Severity**: Medium (accessibility)

**What was the issue**: The gaming and about pages used `<h3>` for all section headings (Overview, Recent Games, Most Played, etc.) directly under the `<h1>` page title, skipping `<h2>` entirely.

**Why it was an issue**: Screen readers and assistive technologies use heading levels to build a navigable document outline. Skipping `<h2>` breaks this outline — users navigating by headings would see a flat list of `<h3>` tags with no intermediate structure. This is a WCAG 2.1 Level A violation (1.3.1 Info and Relationships).

**How it was solved**: Changed 9 `<h3>` tags to `<h2>` — 5 in `gaming.astro` and 4 in `about.astro`. Updated all matching closing tags.

**How it is now**: Clean heading hierarchy: `<h1>` (page title) → `<h2>` (section headings). Screen readers can navigate the document outline correctly.

**Files changed**: `src/pages/gaming.astro`, `src/pages/about.astro`

## 12. Tooltips Not Keyboard-Accessible

**Severity**: Medium (accessibility)

**What was the issue**: Three interactive tooltip tiles (achievements, live sites, hours played) only showed their tooltips on `:hover`. Keyboard users couldn't trigger them.

**Why it was an issue**: Users who navigate with Tab (keyboard-only users, screen reader users, users with motor disabilities) could never see the tooltip content. The tiles also lacked ARIA attributes, so screen readers didn't announce them as interactive elements.

**How it was solved**: Added `tabindex="0"` (makes them focusable), `role="button"` (announces them as interactive), and `aria-label` (describes their purpose) to all three tiles. Added `:focus-within` CSS rules for all three tooltip types so they appear on keyboard focus, not just mouse hover.

**How it is now**: All tooltips are reachable via Tab and visible on focus. Screen readers announce them as buttons with descriptive labels.

**Files changed**: `src/pages/index.astro`, `src/pages/gaming.astro`, `src/styles/global.css`

## 13. Mobile Menu Missing aria-expanded

**Severity**: Medium (accessibility)

**What was the issue**: The hamburger menu button toggled the mobile nav, but didn't set `aria-expanded` to reflect whether the menu was open or closed.

**Why it was an issue**: Screen readers announce `aria-expanded="true/false"` as "expanded" or "collapsed". Without it, a blind user pressing the menu button has no feedback about whether the menu opened or closed.

**How it was solved**: Added `menuBtn.setAttribute('aria-expanded', String(!open))` to the toggle handler, synchronized with the actual open/close state.

**How it is now**: Screen readers correctly announce the menu state on each toggle.

**Files changed**: `src/components/Header.astro`

## 14. Mobile Menu Used Inline style.display

**Severity**: Medium (accessibility / code quality)

**What was the issue**: The mobile menu toggle used `mobileMenu.style.display = 'block'/'none'` — inline styles mixed with CSS class-based styling.

**Why it was an issue**: Inline styles have the highest CSS specificity, making them hard to override. They also don't animate well and create a maintenance split between CSS and JS. If someone added a CSS transition to the menu, the inline style would override it.

**How it was solved**: Switched to CSS class toggling with `mobile-menu-closed` (display: none) and `mobile-menu-open` (display: block) classes. Added these classes to `global.css`.

**How it is now**: Menu visibility is controlled entirely via CSS classes. JS only toggles class names. Animations or transitions can be added in CSS without JS changes.

**Files changed**: `src/components/Header.astro`, `src/styles/global.css`
